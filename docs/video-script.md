## シーン 3: ダッシュボード & KPI 確認（2分30秒）

### ナレーション

> ログインすると、ダッシュボード画面が表示されます。
> ここでは、システム全体の状況を一目で把握できる 6 つの重要指標（KPI）が表示されます。
>
> まず、左上の「総インシデント数」は、これまでに起票されたインシデントの総数です。
> その隣、「オープン件数」は、まだ解決していないインシデントの数。
> 「P1 アクティブ件数」は、最高優先度で、かつ未解決のインシデント数です。
>
> そして、重要なのが SLA に関連した 2 つの指標。
> 「SLA 応答違反件数」は、対応期限を超過したインシデント数。
> 「SLA 解決違反件数」は、解決期限を超過したインシデント数です。
>
> 最後に「平均解決時間」は、インシデント起票から完全解決までの平均時間を表しています。

### 画面表示

#### 3-1: ダッシュボード全体（45秒）

```
┌───────────────────────────────────────────┐
│ ITSM インシデント管理     [👤 operator_001▼]│
├───────────────────────────────────────────┤
│                                           │
│ 📊 KPI                                     │
│ ┌──────────┬──────────┬─────────────────┐ │
│ │ 総件数   │ 開放件数 │ P1 Active      │ │
│ │   45     │    8     │      2         │ │
│ ├──────────┼──────────┼─────────────────┤ │
│ │ 応答違反 │ 解決違反 │ 平均解決時間    │ │
│ │    1     │    3     │   2h 15m       │ │
│ └──────────┴──────────┴─────────────────┘ │
│                                           │
│ ステータスボタン                           │
│ [new] [assigned] [in_progress] ...       │
│                                           │
│ インシデント一覧                           │
│ [+インシデント起票]                        │
│ ┌───────────────────────────────────────┐ │
│ │ ID    │ タイトル    │ 優先度 │ SLA   │ │
│ │ INC-10│ ログイン不可│ 🔴P1 │ 2m   │ │
│ │ INC-09│ DB接続エラー│ 🟡P2 │ 45m  │ │
│ └───────────────────────────────────────┘ │
│                                           │
└───────────────────────────────────────────┘
```

#### 3-2: ステータスボタン説明（45秒）

**ナレーション：**
> その下には、ステータスボタンが表示されています。
> [new] — 未割当のインシデント
> [assigned] — 割り当てられたが未開始
> [in_progress] — 対応中
> [resolved] — 解決済み
> [closed] — クローズ済み
>
> これらのボタンをクリックすれば、そのステータスのインシデントだけを絞り込んで表示できます。

**実演内容：**
1. [new] ボタンをホバー（ハイライト表示）
2. [new] ボタンをクリック
3. テーブルが更新され、「new」ステータスのインシデントのみ表示
4. [in_progress] ボタンをクリック
5. テーブル更新

## シーン 4: インシデント起票（3分）

### ナレーション

> それでは、実際にインシデントを起票してみましょう。
> 「インシデント起票」ボタンをクリックします。

### 画面表示

#### 4-1: 起票ダイアログ表示（10秒）

```
┌────────────────────────────────────┐
│ インシデント起票                    │
├────────────────────────────────────┤
│                                    │
│ タイトル *                          │
│ [__________________________]        │
│                                    │
│ 説明 *                              │
│ ┌──────────────────────────────┐   │
│ │                              │   │
│ └──────────────────────────────┘   │
│                                    │
│ 優先度                              │
│ [▼ P3]                             │
│                                    │
│ 担当者                              │
│ [▼ 未指定]                         │
│                                    │
│ [キャンセル] [起票]                 │
│                                    │
└────────────────────────────────────┘
```

#### 4-2: 入力実演（100秒）

**ナレーション：**
> ダイアログが開きました。
> まずは「タイトル」にインシデント概要を入力します。
> 例えば、「ログインシステムが応答しない」と入力します。
>
> 次に「説明」欄に、より詳しい状況を記入します。
> 「午前 9 時ごろからログインが失敗するようになりました」。
>
> その下の「優先度」は、このインシデントの緊急度を選択します。
> P1（Critical） — システム全体停止など最高優先度
> P2（High） — 主機能が利用不可
> P3（Medium） — 機能の一部が利用不可
> P4（Low） — 軽微な不具合
>
> 今回は「P1」を選択します。
>
> 「担当者」は、このインシデントに対応するメンバーを指定します。
> ここでは「operator_001」を選択しましょう。
>
> すべての項目を入力したら、「起票」ボタンをクリック。

**実演内容：**
1. タイトル欄をクリック
2. 入力：`ログインシステムが応答しない`
3. 説明欄をクリック
4. 入力：`午前9時ごろからログインが失敗するようになりました`
5. 優先度ドロップダウンをクリック
6. `P1` を選択
7. 担当者ドロップダウンをクリック
8. `operator_001` を選択
9. 「起票」ボタンをクリック
10. ローディング表示
11. インシデント詳細画面に遷移

#### 4-3: 起票完了画面（30秒）

**ナレーション：**
> インシデントが起票されました。
> 見てください。システムが自動的にインシデント ID「INC-0010」を採番しています。
> 
> さらに、Slack に通知が送信されました。
> （Slack 通知スクリーンショット表示）
> 優先度絵文字、インシデント ID、タイトル、起票者が記載されています。
> 管理者やチームメンバーが即座に状況を把握できるわけです。

**画面表示：**
- インシデント詳細画面
- 詳細欄に以下が表示：
  - ID: INC-0010
  - タイトル: ログインシステムが応答しない
  - 優先度: 🔴 P1
  - ステータス: new
  - 担当者: operator_001
  - SLA 応答期限: 15 分
  - SLA 解決期限: 4 時間

## シーン 5: インシデント詳細 & 更新（3分30秒）

### ナレーション

> インシデント詳細画面を詳しく見ていきましょう。
> 画面上部には、インシデント ID、優先度バッジ、ステータス、SLA 状態が一目で確認できます。

### 画面表示

#### 5-1: 詳細画面構成説明（45秒）

```
画面ヘッダー表示:
INC-0010 [🔴 P1] [new] [⚠️ 要注意] [期限内]

基本情報:
タイトル: ログインシステムが応答しない
起票者: admin | 起票日: 2026/05/09 09:15

SLA 詳細:
応答期限: 14m 30s | 応答日時: -
解決期限: 3h 59m | 解決日時: -
```

**ナレーション：**
> まず、SLA バッジに注目してください。
> 「14m 30s」と表示されているのは、応答期限までの残り時間です。
> 緑色で表示されているのは、「まだ時間に余裕がある」という意味。
>
> もし、この時間が 30 分未満になると、色がオレンジに変わり、警告を促します。
> さらに期限を超過すると、赤色になって「SLA 違反」を示します。

#### 5-2: ステータス変更実演（60秒）

**ナレーション：**
> では、このインシデントの対応を開始します。
> ステータスボタンをクリックして、ステータスを変更しましょう。

**実演内容：**
1. ステータスボタン `[new]` をクリック
2. ドロップダウン表示：
   ```
   [ assigned ]
   [ in_progress ]
   [ closed ]
   ```
3. `[ in_progress ]` をクリック
4. ステータスが `new` → `in_progress` に変更
5. タイムラインに以下が記録：
   ```
   [09:45] operator_001 がステータスを new → in_progress に変更
   ```
6. Slack 通知送信（画面に表示）

#### 5-3: コメント追加実演（45秒）

**ナレーション：**
> インシデント対応中は、コメントを追加して進捗を共有することが大事です。
> 下部のコメント入力欄を使いましょう。

**実演内容：**
1. コメント入力欄をクリック
2. 入力：`対応を開始しました。ログシステムを再起動します`
3. 「コメント送信」ボタンをクリック
4. タイムラインに記録表示
5. コメント入力欄がリセット

#### 5-4: タイムライン表示（30秒）

**ナレーション：**
> 注目してください。タイムラインには、すべての変更履歴とコメントが、新しい順に表示されています。
> このように、「誰が、いつ、何をしたのか」が完全に記録されるので、監査対応にも役立ちます。

## シーン 6: インシデント解決 & SLA 達成（2分）

### ナレーション

> インシデント対応が完了したとします。
> ステータスを「resolved」に変更して、解決を記録します。

### 画面表示

#### 6-1: 解決ステータス変更実演（45秒）

**実演内容：**
1. ステータスボタン `[in_progress]` をクリック
2. ドロップダウン表示
3. `[ resolved ]` をクリック
4. ステータス変更：`in_progress` → `resolved`
5. 解決日時が自動記録
6. SLA バッジが「SLA 達成」に変更

#### 6-2: SLA 達成表示（45秒）

**ナレーション：**
> 見てください。SLA 解決バッジが「SLA 達成」に変わりました。
> これは、解決期限（この場合 4 時間）内にインシデントが解決したことを意味します。
>
> もし、この期限を超えていたら、赤色で「SLA 違反」と表示されていたはずです。
> ダッシュボードの「SLA 解決違反件数」にもカウントされていたでしょう。

**画面表示：**
- SLA バッジが「SLA 達成」に更新
- タイムラインに「resolved」への変更が記録
- Slack 通知が送信される

## まとめ

本スクリプトを参考に、実際の画面をキャプチャして動画化することで、ユーザーフレンドリーな解説動画が完成します。

- **シーン総数：** 7 シーン（計 15～20 分）
- **対象者：** 一般オペレータ、管理者、管理層
- **目的：** 新規ユーザーのオンボーディング、機能紹介、運用マニュアル

各シーンは独立して、ユースケース別に分割・再構成することもできます。
